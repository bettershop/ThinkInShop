# 单体架构说明篇（laike-apis）

## 1. 架构目标

本架构文档说明 `thinkinshop` 在“单体部署模式”下的设计与运行方式，目标是：

1. 仅发布一个应用进程：`laike-apis`。
2. 保持业务域模块化（代码分层），但运行时进程单体化。
3. 支持快速上线、可观测、可回滚。

## 2. 单体模式的模块关系

`laike-apis` 是统一 API 入口模块，在运行时聚合以下能力：

- `laike-admins`：管理端业务接口
- `laike-apps`：买家端业务接口
- `laike-comps`：公共组件能力
- `laike-plugins`：营销插件能力
- `laike-root`：网关工具、基础服务
- `laike-common / laike-domain / laike-core`：基础组件与领域模型

逻辑上是“多模块”，物理上是“单进程”。

## 3. 运行时拓扑

```text
[Client/H5/App/Admin]
        |
        v
[SLB/Nginx/Ingress]
        |
        v
[laike-apis 单体实例]
   |            |            |
   v            v            v
[MySQL]      [Redis]      [对象存储/第三方]
```

## 4. 请求链路与职责分层

### 4.1 请求路径

1. 外部请求进入负载均衡（Nginx/Ingress）。
2. 转发到 `laike-apis` 实例（默认端口 `21898`，上下文 `/apis`）。
3. Controller 层分发到对应模块 Service。
4. Service 调用 `mapper`/`repository` 访问 MySQL、Redis。
5. 结果统一返回。

### 4.2 代码分层建议

- 表现层：`*Controller`
- 应用服务层：`*Service` / `*ServiceImpl`
- 数据访问层：`*Mapper`
- 公共层：`laike-root` + `laike-common`

收益：

1. 代码可维护性高。
2. 单体形态下调试链路短。
3. 未来可按模块平滑拆分微服务。

## 5. 配置架构

### 5.1 当前默认行为

`laike-apis` 的 `bootstrap.yml` 中已关闭 Nacos Config 与 Discovery，因此单体模式下可直接使用本地/外置配置启动。

### 5.2 推荐配置优先级

1. 命令行参数（最高优先级）
2. 外置配置文件（`--spring.config.additional-location`）
3. 包内 `bootstrap.yml`（最低）

### 5.3 配置治理原则

1. 敏感信息（DB 密码、密钥）只放 Secret 或环境变量。
2. 环境差异化参数全部外置化。
3. 应用镜像保持“同镜像多环境”。

## 6. 数据与中间件依赖

单体模式关键依赖：

- MySQL：交易、商品、会员、营销等主数据
- Redis：缓存、会话、热点数据
- 对象存储：图片/文件
- XXL-JOB（可选）：定时任务

建议：

1. MySQL 开启主从与备份策略。
2. Redis 使用哨兵或集群，避免单点。
3. 对象存储配置独立账号与最小权限。

## 7. 可用性设计

### 7.1 Docker 场景

- 最少 `2` 个容器副本，前挂 Nginx 负载。
- 容器重启策略 `always`。
- 日志与上传目录持久化挂载。

### 7.2 K8s 场景

- Deployment 副本数 `>=2`。
- 配置 `livenessProbe/readinessProbe`。
- HPA 基于 CPU 和内存自动扩缩容。
- 使用滚动升级与版本回滚。

## 8. 性能与容量规划

### 8.1 起步容量

- QPS < 100：1~2 实例，2C4G
- QPS 100~300：2~4 实例，4C8G
- 高频大促：建议压测后按 RT 与 CPU 线性扩容

### 8.2 JVM 建议

- `-Xms` = `-Xmx`
- 使用 `G1GC`
- 开启 GC 日志并纳入观测

### 8.3 数据库建议

1. 慢 SQL 监控（按模块维度定位）。
2. 热表索引审计。
3. 读多写少场景可引入读写分离。

## 9. 安全基线

1. 全链路 HTTPS。
2. 出入口网络白名单。
3. 管理接口做 IP 限制与鉴权。
4. 关键配置与证书不落代码仓库。
5. 容器镜像定期扫描漏洞。

## 10. 发布与回滚架构

### 10.1 发布策略

1. 构建唯一版本镜像（不可变 tag）。
2. 先预发验证，再生产滚动。
3. 发布窗口内重点观察错误率与响应时间。

### 10.2 回滚策略

- Docker：快速切回上一稳定镜像 tag。
- K8s：`kubectl rollout undo deploy/laike-apis`。
- 数据变更：DDL 必须前置兼容，避免应用回滚失败。

## 11. 从单体到微服务的演进路径

建议按业务边界逐步拆分，而不是一次性重构：

1. 先把配置、日志、链路追踪标准化。
2. 从低耦合模块（如营销插件）优先拆分。
3. 引入网关统一鉴权与路由。
4. 数据层保持“先兼容、再迁移”。

这样可以保持持续交付能力，同时降低架构迁移风险。
