# 集群架构说明篇（高可用与水平扩展）

## 1. 架构原则

`thinkinshop` 采用“单体应用 + 云中间件”架构：

1. 应用统一入口仍是 `laike-apis`。
2. 运行形态从单实例升级为多实例集群。
3. 状态与资源外置到共享基础设施。

核心目标：

- 可横向扩展
- 可滚动发布
- 可快速回滚
- 可统一观测（日志/监控）

## 2. 逻辑与物理解耦

### 2.1 逻辑层（代码）

- `laike-apis` 聚合 `admins/apps/comps/plugins/root/common/domain`。
- 业务模块仍按领域划分，便于维护。

### 2.2 物理层（运行）

- 多个 `laike-apis` 实例对外提供服务。
- 所有实例共享同一套 RDS、Redis、OSS、SLS。

## 3. 核心依赖分工

1. `RDS`：事务主数据（订单、商品、用户、配置）。
2. `Redis Cluster`：会话、缓存、分布式锁、跨节点消息分发。
3. `OSS`：图片与对象存储，避免本地盘分裂。
4. `SLS`：集中日志检索、告警、审计。

## 4. 高可用设计点

### 4.1 应用层

1. 多副本部署（ECS 多机或 K8s 多 Pod）。
2. 健康检查驱动流量摘除。
3. 滚动升级避免整体中断。

### 4.2 数据层

1. RDS 使用高可用架构。
2. Redis 使用 Cluster（建议主从+故障转移）。
3. 数据库与缓存均在同 VPC 低延迟互联。

### 4.3 存储与日志

1. 文件上 OSS，解决本地盘一致性问题。
2. 日志入 SLS，避免节点漂移导致日志丢失。

## 5. WebSocket 集群化设计

### 5.1 旧问题

旧实现只维护本机 `webSocketMap`，消息只能发给当前节点连接；多实例时会出现跨节点不可达。

### 5.2 新方案

新增“本地投递 + Redis Pub/Sub”双通道：

1. 优先本机投递（低时延）。
2. 本机未命中时，发布 Redis 频道。
3. 其他节点消费后向本地连接投递。

收益：

- 支持 SLB/Ingress 后端多实例。
- 不依赖强制粘性会话。
- 不改业务接口与消息模型。

## 6. 扩缩容模型

### 6.1 触发依据

1. CPU 使用率
2. JVM 堆使用率
3. 接口 RT/P99
4. WebSocket 在线连接数

### 6.2 扩容策略

1. 常态最少 2 副本。
2. 高峰按 HPA 自动扩容。
3. Redis/RDS 规格与连接池同步扩容。

## 7. 可观测性与治理

### 7.1 日志

1. 业务日志、错误日志统一采集到 SLS。
2. 建议字段：`traceId`、`service`、`storeId`、`userId`。

### 7.2 监控

1. 暴露 `actuator` 健康与指标。
2. 监控 JVM、线程池、慢 SQL、Redis 延迟。

### 7.3 告警

1. 接口错误率告警。
2. WebSocket 连接数异常波动告警。
3. Redis 连接池耗尽告警。

## 8. 安全与隔离

1. 所有凭据放 Secret/环境变量，不落仓库。
2. 仅开放 SLB/Ingress 公网入口端口。
3. 内部资源（RDS/Redis/OSS）按白名单最小权限访问。

## 9. 发布与回滚架构

### 9.1 发布

1. 构建不可变镜像标签。
2. 灰度/分批滚动发布。
3. 观察关键指标后再全量。

### 9.2 回滚

1. 回滚镜像版本。
2. 保证 DB 变更向前兼容。
3. 回滚后验证健康检查与关键链路。
